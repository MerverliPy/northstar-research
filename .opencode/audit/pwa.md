| `navigateFallback` | Not configured | ⚠️ Falls back to implicit precache behavior |
| `offlineFallback` / `offlinePage` | Not configured | ⚠️ No dedicated offline page HTML |

### 2.2 Runtime Caching

| Pattern | Strategy | Cache Name | Max Entries | Max Age | Assessment |
|---------|----------|------------|-------------|---------|------------|
| `/api/.*` | NetworkFirst | `api-cache` | 100 | 300s (5m) | ⚠️ See notes below |

**API caching concerns:**
- `NetworkFirst` is correct for API GET requests
- BUT the regex `/^\/api\/.*/i` matches ALL API calls including POST/PUT/DELETE mutations
- Workbox only caches successful (200) GET requests by default, so mutations are safe
- 5-minute cache duration is very short — limits offline API capability
- No separate caching strategy for static API responses (e.g., settings, entity lists)

### 2.3 Precache Manifest (generated `sw.js`)

The generated precache includes 22 entries: `index.html`, `favicon.svg`, `icons.svg`, 5 icon PNGs, the main JS bundle (`index-C5JxSVDA.js`, ~300 KB), CSS (`index-BMihmTlc.css`, ~37 KB), 11 lazy-loaded chunks, and the Workbox runtime. **Total precache payload: ~1.1 MB.**

### 2.4 Registration & Lifecycle (`usePWA.ts`)

```typescript
const wb = new Workbox('/sw.js')  // Manual WorkboxWindow instance
wb.addEventListener('waiting', () => { /* toast with reload button */ })
wb.addEventListener('activated', () => { /* success toast */ })
wb.register()
```

| Aspect | Status | Notes |
|--------|--------|-------|
| Manual registration | ✅ | Via `usePWA()` hook in `<App>` |
| Update detection | ✅ | `waiting` event → toast with "Reload" action |
| Update UX | ✅ | User-visible toast; calls `wb.messageSkipWaiting()` then `window.location.reload()` |
| Update after activation | ✅ | Success toast on `activated` |
| Error handling | ⚠️ | No `onerror`/`redundant` handler — silent SW failures |

### Service Worker Severity Findings

- **HIGH (4):** No explicit navigation fallback configuration — offline navigation behavior depends on implicit Workbox defaults; can cause inconsistent app-shell serving
- **MEDIUM (5):** API cache uses 300s expiration — makes offline API capability very limited
- **MEDIUM (6):** No SW error/redundant lifecycle event handling — SW failures are silent
- **MEDIUM (7):** API runtime cache pattern matches mutation endpoints (POST/PUT/DELETE) — safe for now (Workbox only caches GET) but fragile if Workbox behavior changes
- **LOW (8):** `includeAssets` is redundant with `globPatterns`
- **LOW (9):** `useEffect` cleanup only posts `SKIP_WAITING` via raw message; should use `wb.messageSkipWaiting()` for consistency

## 7. Portal-Specific Integration Issues

### 7.1 Route Collision: Jinja2 vs SPA

**CRITICAL (24):** FastAPI route collision between Jinja2 template routes and SPA client-side routes.

```python
# main.py — Jinja2 routes registered BEFORE SPA catch-all:
app.include_router(dashboard.router, prefix="/dashboard")   # Jinja2
app.include_router(cleanup.router, prefix="/cleanup")       # Jinja2
app.include_router(extraction.router, prefix="/extraction") # Jinja2
app.include_router(quality.router, prefix="/quality")       # Jinja2
app.include_router(visual.router, prefix="/visual")         # Jinja2

@app.get("/{full_path:path}")                               # SPA catch-all
async def serve_spa(full_path: str):
```

**Impact:**
- Page reload at `/dashboard` → serves Jinja2 template (no PWA, no SW, no manifest link)
- Page reload at `/cleanup` → serves Jinja2 template
- The SPA client-side routes at the same paths are unreachable on full page loads
- `base.html` (Jinja2) links point to conflicting routes (e.g., `<a href="/graph">` → would serve SPA, but visual router is at `/visual/`)

**Resolution required:**
Either (a) move Jinja2 routes to distinct prefixes (e.g., `/legacy/dashboard`, `/legacy/cleanup`), or (b) have the Jinja2 routes check for SPA static files and redirect, or (c) serve the SPA index.html for all unknown routes and only serve Jinja2 templates when SPA hasn't been built.

### 7.2 Jinja2 Templates Lack PWA Support

**HIGH (25):** The 6 Jinja2 templates (`base.html`, `dashboard.html`, `cleanup.html`, `extraction.html`, `quality.html`, `graph_viewer.html`) have **zero PWA meta tags:**
- No `<link rel="manifest">`
- No `<meta name="theme-color">`
- No `apple-mobile-web-app-*` tags
- No `viewport-fit=cover`
- No service worker registration
- No dark theme inline script
- External CDN dependencies (htmx, alpinejs, vis-network via CDN)

When a user lands on a Jinja2-rendered page (via full page load at `/dashboard`), they get no PWA capabilities whatsoever.

### 7.3 Static File Serving Issues

**MEDIUM (26):** `favicon.svg` mount exposes entire `STATIC_DIR`:
```python
app.mount("/favicon.svg", StaticFiles(directory=str(STATIC_DIR)), name="spa_favicon")
```
This makes `/favicon.svg/assets/...`, `/favicon.svg/manifest.webmanifest`, `/favicon.svg/sw.js` all accessible. While not a security vulnerability (these files are meant to be public), it's a misconfiguration that could cause confusion.

**LOW (27):** Static file serving only checks for `assets/` directory existence at startup. If the SPA hasn't been built and `static/assets/` doesn't exist, no SPA routes work — all navigation hits the Jinja2 fallback or 404.

### 7.4 SPA Fallback Routing Assessment

**✅ Good:** Path traversal protection is correctly implemented:
```python
file_path = (STATIC_DIR / full_path).resolve()
if not str(file_path).startswith(str(STATIC_DIR.resolve())):
    raise HTTPException(status_code=404)
```

**✅ Good:** Fallback chain is: static file → SPA index.html → Jinja2 base template → 404.

**⚠️ Issue:** The catch-all `/graph/data/{project_id}` route (line 95-105 in `main.py`) shadows the visual router's `/visual/data/{project_id}` route, which could cause confusion.

## 11. Summary of Top 5 Findings

### 1. CRITICAL — Route Collision: Jinja2 Templates vs SPA (F24)
The FastAPI server registers Jinja2 template routes at `/dashboard`, `/cleanup`, `/extraction`, `/quality` **before** the SPA catch-all. Page reloads at these paths serve non-PWA Jinja2 pages instead of the SPA. This breaks the PWA experience on these routes and creates an inconsistent dual-interface system.

### 2. HIGH — Missing 512×512 Maskable Icon (F1)
Android requires a 512×512 maskable icon for adaptive icons on high-DPI launchers. Without it, the app icon appears in a white circle on modern Android launchers, degrading the installed app experience.

### 3. HIGH — No Offline Fallback Page (F10)
Workbox is not configured with `offlineFallback` or `navigateFallback` options. If a user navigates to an uncached route while offline, they may see a browser error page instead of the app shell. The implicit precache fallback to `index.html` is fragile.

### 4. HIGH — Jinja2 Templates Have Zero PWA Support (F25)
All 6 Jinja2 templates lack `<link rel="manifest">`, theme-color meta tags, apple-mobile-web-app tags, `viewport-fit=cover`, and service worker registration. Pages served via these templates are never treated as PWA pages by browsers.

### 5. MEDIUM — No In-App Navigation for Standalone Mode (F17)
When running in `display: standalone` mode, there is no browser back button, refresh button, or address bar. The app provides no in-app back navigation, breadcrumbs, or pull-to-refresh. Users who navigate deep into the app have no way to go back without using the sidebar.
